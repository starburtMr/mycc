本文件中的“全局执行协议”为全局高优先级规则；“版本控制与原子提交约束（强制）”以 `0-System/about-me/persona.md` 为唯一真源，适用于所有任务与所有项目目录。

- 开发规范统一真源：`0-System/rules/ENGINEERING_CONSTITUTION.md`（人类可读）与 `0-System/rules/engineering-policy.yaml`（机器可读）。

## 全局执行协议（强制）

### A. 总体行为准则

1. 不臆测：若需求/上下文缺失会影响正确实现，必须先提出最少且关键的问题；在用户补充前，仅可输出：
   - 需要补充的信息清单
   - 不同假设下的可选方案对比
   - 不得直接输出最终实现
2. 不编造：不得编造不存在的文件、API、配置、命令、日志结果、依赖版本；不确定时必须明确标注“需要确认/无法从上下文得知”。
3. 最小改动：优先在现有结构内小步修改，避免无必要重构；若建议重构，必须说明收益、风险与低风险替代方案。
4. 可交付：代码必须可运行（或可被集成），并提供最小示例与测试/验证方式。
5. 先方案后代码：除非用户明确要求“直接给代码”，否则先给简短方案再实现。
6. 安全默认：对外部输入零信任，避免注入、越权、敏感信息泄露；示例不得包含真实密钥。

### B. 工程硬约束

【可读性】
- 命名必须语义化：变量名词、函数/方法动词短语、类名词。
- 禁止无意义命名（如 `a/b/tmp/foo/data1`，极短循环索引 `i/j` 除外）。
- 遵循语言主流规范（Python PEP8、JS/TS 社区规范、Go fmt 等）。

【单一职责】
- 函数单一职责；复杂流程拆分为清晰函数。
- 单函数建议 <= 30 行；超过需说明并保持可读。
- 参数建议 <= 5 个；超过建议使用对象/结构体封装。

【DRY 与抽象】
- 重复逻辑必须抽取复用。
- 抽象“刚好够用”，避免过度工程（YAGNI）。

【错误处理】
- 禁止吞异常（如 `except: pass`）。
- 捕获具体异常类型，返回/抛出可诊断信息。
- 错误信息与日志不得泄露敏感数据。

【输入校验】
- 所有外部输入必须校验（HTTP、文件、环境变量、CLI、DB 不可信字段）。
- 校验失败需明确报错（含错误码，若适用）。

【边界与状态】
- 避免全局可变状态，优先参数传递/依赖注入/对象封装。
- 禁止魔法数字/字符串，必须提取命名常量（如 `MAX_RETRY = 3`）。

【测试与可维护】
- 核心业务逻辑必须可单元测试，尽量业务与 IO 分层。
- 覆盖关键路径、边界条件、异常路径。
- 变更必须附回归测试建议或新增测试。

### C. 安全与质量门槛

- SQL/命令/路径禁止拼接：SQL 参数化、命令参数数组、路径安全 join 并校验。
- 避免不安全反序列化、`eval`、任意模板执行。
- 敏感信息只读安全配置；示例使用占位符；日志禁止敏感字段。
- 并发/异步避免竞态；共享资源加锁或使用线程安全结构，并说明线程安全假设。
- 性能避免明显低效（N+1、无界循环、重复 IO）；必要时给复杂度与优化点。

### D. 输出格式（默认）

在通用技术说明/代码交付场景下，输出顺序默认使用：
1. 需求理解（<= 8 行）
2. 关键澄清问题（如无写“无”）
3. 设计方案（含取舍）
4. 代码实现（优先 diff）
5. 使用示例（最小可运行）
6. 测试（单元/集成 + 命令）
7. 自检清单（正确性/边界/错误处理/安全/性能/可读性/测试）

### E. Diff 与文件规则

- 若存在现有代码，必须基于现有代码输出统一 diff。
- 只改必要部分，禁止粘贴无关代码。
- 新增文件必须说明用途与路径。
- 配置变更必须说明迁移方式与默认值。

### F. 互动与迭代

- 反馈报错/失败时，按顺序处理：
  1. 复述现象
  2. 按概率给出原因列表
  3. 给验证步骤
  4. 给修复 diff
- 多方案时给 2~3 个方案，对比优缺点与适用场景，并给推荐。

### G. 语言附加规则（按需启用）

- Python：类型注解、ruff/black 风格、保留异常链、避免可变默认参数。
- JS/TS：优先 TypeScript、严格类型、避免 `any`、异步错误显式处理。
- Java：不可变优先、异常分层、日志用 slf4j。
- Go：显式 `err`、传递 `context`、避免 `panic`、`go fmt`。

### H. 终止条件（模板要求）

- 当任务完成且为“完整交付型输出”时，末行使用：
  “已按约束输出：可运行代码 + 示例 + 测试 + 自检清单”

## AI 工程实践补充约束（全局强制）

### 1) 小步开发与逐步增强

- 先交付最小可运行版本（MVP），再迭代增强功能、错误处理、测试与优化。
- 禁止一次性实现跨多个子系统的大而全改动。

### 2) 每一步都必须可运行

- 每个步骤完成后必须满足：可编译、可运行、可验证（至少通过已有关键测试或最小验收检查）。
- 禁止“先改很多模块，最后一起修”。

### 3) 显式复杂度管理

- 复杂逻辑必须封装为语义化函数/模块，不得以内联条件堆叠隐藏复杂度。
- 当改动超过 2 个模块或预计超过 200 行，必须先给设计（模块结构、数据流、风险）再编码。

### 4) 依赖治理

- 新增依赖必须显式说明：名称、版本、用途、替代方案与风险。
- 禁止隐式引入依赖或在无说明情况下升级核心依赖。

### 5) 配置与代码分离

- 环境相关参数必须来自配置/环境变量，不得硬编码生产地址、密钥或环境常量。

### 6) IO 与业务逻辑分层

- 业务逻辑应尽量纯函数化；IO（数据库/网络/文件）放在边界层。
- 禁止在核心计算函数中直接耦合数据库或外部调用。

### 7) 接口契约完整性

- 新增/变更 API 必须明确：输入 schema、输出 schema、错误码与兼容策略。
- 破坏性变更必须单独步骤并附迁移方案。

### 8) 防御式编程

- 默认输入不可信、网络不可靠、依赖会失败。
- 必须配置合理的 timeout、重试策略（如适用）与错误处理路径。

### 9) 结构化日志与可调试性

- 服务端关键路径日志必须结构化，至少包含 `request_id` 与关键业务 ID（如 user_id/order_id/task_id）。
- 错误信息可诊断但不泄露敏感信息；禁止吞异常与静默失败。

### 10) 简单优先原则

- 每次编码前必须先检查是否有更简单实现：`simple > clever`。
- 优先删除冗余与重复，减少无价值抽象，避免过度设计。

### 11) 模块职责声明

- 新增模块/文件必须说明“职责边界”（它负责什么、不负责什么）。

### 12) 算法复杂度说明（按需）

- 涉及算法或高复杂度处理时，必须给出时间复杂度与空间复杂度，并说明瓶颈与可优化点。

## AI Coding Constitution v1（全局强制）

> 说明：本节为工程级硬约束，和上文规则并行生效；冲突时以“更保守、更可回滚、更可测试”的规则为准。

### I. 需求与设计

1. 不允许猜需求：需求不足会影响正确性时，必须先澄清，不得直接产出最终实现。
2. 复杂功能先设计：满足任一条件（>200 行、>2 模块、涉及数据结构、涉及系统交互）必须先给设计（模块划分、数据流、接口定义）。
3. 设计优先简单：`Simple > Clever`，避免过度抽象与不必要层级。

### II. 编码原则

4. 可读性优先：命名语义化，禁止无意义命名。
5. 单一职责：函数建议 <=30 行、<=5 参数、只做一件事。
6. DRY：重复逻辑必须抽取，禁止复制粘贴。
7. 禁止魔法数字/字符串：提取命名常量。
8. 避免全局可变状态：优先参数传递、对象封装、依赖注入。
9. IO 与业务逻辑分离：核心业务逻辑尽量纯函数化。
10. 配置与代码分离：环境参数来自配置/环境变量，禁止硬编码环境地址与密钥。

### III. 安全与鲁棒性

11. 输入必须校验：HTTP、CLI、文件、环境变量、DB 不可信字段均需验证。
12. 错误处理明确：禁止吞异常，捕获具体异常并提供可诊断信息。
13. 防御式编程：默认输入不可信、网络不可靠、依赖会失败；落实 timeout/retry/fallback。
14. 禁止危险操作：禁止 `eval`、动态 SQL 拼接、命令拼接。

### IV. 可维护性

15. 模块职责清晰：新增模块必须声明职责边界。
16. 控制嵌套：超过 3 层嵌套必须重构。
17. 删除优先：优先消除冗余，避免无价值抽象叠加。
18. 可调试性：日志可追踪、错误可定位、不隐藏异常。

### V. 测试

19. 核心逻辑必须可单测：减少对 IO 的强耦合。
20. 测试必须覆盖：正常路径、边界条件、异常路径。

### VI. 版本控制

21. 原子提交：一次提交只做一类改动（功能/修复/重构/格式化）。
22. 大改动必须拆步：每步可运行、可测试、可回滚。
23. 提交信息使用 Conventional Commits：`feat/fix/refactor/test/docs/chore`。

### VII. 可回滚性

24. 每个改动必须说明回滚方法（如 `git revert <commit>` 或步骤化回退方案）。

### VIII. 输出质量

25. 交付型输出默认包含：需求理解、设计方案、代码实现、使用示例、测试、自检清单。

### 宪法级附加条款（必须）

- 若存在更简单实现，必须优先使用简单实现。
- 输出前必须自检：
  1. 是否存在重复代码
  2. 是否有未处理异常
  3. 是否有魔法数字/字符串
  4. 是否可测试
  5. 是否可读

## 会话与执行约束

- Codex 可写共享区域与 `.codex/`。
- 未经用户明确要求，Codex 不得修改 `.claude/`。
- 每次会话开始前，先读取：`SOUL.md`、`persona.md`、`user-profile.md`、`tasks/index.md`、当前任务文件、项目 `context/`。
- 不依赖“脑内记忆”，关键信息必须落盘到任务或项目文档。
- `tasks/` 是任务状态唯一真源。
- 每次任务状态迁移都必须同步到 `tasks/`。
- 任务关键字段不得填写占位值（如 `TBD`、`todo`、`pending`、`none`）。
- 技术执行类任务默认先自主尝试 5 次再中断提问。
- 例外中断条件：权限/密钥缺失、外部服务不可用、风险过高、token 成本超阈值。
- 会话上下文达到 60% 时，必须进行阶段摘要压缩。
- 压缩结果必须回写到任务记录和 `0-System/status.md`。
- 每次路由必须记录 `route_reason`（为何选该模型）。

## Skill 创建约定（系统级）

- 本系统后续创建任何 Skill（不区分 Claude/Codex）都必须先执行：`bash scripts/skills/init-skill.sh <skill-name>`。
- Skill 目录必须统一放在：`skills-core/skills/<skill-name>/`。
- 每个 Skill 必须包含：`SKILL.md`，以及四个子目录（允许为空）：`agents/`、`scripts/`、`references/`、`assets/`。
- 新 Skill 默认进入隔离流程：`quarantine -> promote -> register -> verified`。
- `lifecycle_status != verified` 的 Skill 禁止默认路由（`routing.default_enabled` 必须为 `false`）。
- 每个 Skill 必须声明 `governance.blast_radius`（`readonly|project_write|external_side_effect`）。
- 创建后必须注册到 `skills-core/skill-registry.yaml`，并通过：
  - `bash scripts/guardrails/check_skills_consistency.sh`
  - `bash scripts/guardrails/check_skill_structure.sh`
- 禁止手工绕过脚手架直接创建不合规目录结构。

## EvoMap 只读规则

- EvoMap 仅用于查询经验，不用于注册节点或接取任务。

- 允许调用：知识查询接口（当前仅 `POST /a2a/skill/search`）与公开文档读取。
