# ENGINEERING CONSTITUTION

> 用途：作为蜂巢并行开发的人类可读规范真源。  
> 适用对象：Claude、Codex、子代理、人工协作者。  
> 冲突处理：若与其他规则冲突，以本文件与 `engineering-policy.yaml` 中更严格者为准。

## 1. 核心目标

1. 交付可运行、可测试、可维护、可回滚的工程结果。
2. 默认安全优先：输入不可信、网络不可靠、依赖可能失败。
3. 复杂任务先设计，避免一次性大改。

## 2. 原子变更与分步交付

1. 每次改动必须原子化：可独立理解、可独立测试、可独立回滚。
2. 禁止在同一步混合：功能开发、重构、格式化、依赖升级、无关清理。
3. 大改动必须分步（Step 1/2/3），每步都要有：
- 目的
- 影响范围
- 风险
- 验证
- 回滚

## 3. 编码与结构原则

1. `simple > clever`：优先简单实现，避免炫技代码。
2. 单一职责：函数只做一件事；复杂流程拆分。
3. DRY：重复逻辑必须抽取。
4. IO 与业务分层：核心逻辑尽量纯函数化，可单测。
5. 禁止魔法数字/字符串：提取命名常量。

## 4. 安全与鲁棒性

1. 所有外部输入必须校验（HTTP/CLI/文件/环境变量/DB不可信字段）。
2. 禁止 SQL 拼接、命令拼接、`eval`、危险反序列化。
3. 禁止吞异常；错误必须可诊断且不泄露敏感信息。
4. 默认考虑 timeout/retry/fallback。

## 5. 依赖与配置

1. 新增依赖必须说明：名称、版本、用途、替代方案、风险。
2. 配置与代码分离：环境参数来自配置或环境变量。
3. 禁止硬编码生产地址、密钥、token、cookie。

## 6. 测试与验证

1. 核心逻辑必须可单元测试。
2. 变更至少覆盖：正常路径、边界条件、异常路径。
3. 每步改动后必须至少通过最小验收（可运行/可验证）。

## 7. 提交与回滚

1. 提交信息必须使用 Conventional Commits：
- feat
- fix
- refactor
- test
- docs
- chore
2. 每步都必须有回滚方案（如 `git revert <commit>` 或步骤化回退）。
3. 破坏性变更必须单独一步，并提供迁移方案。

## 8. 并行蜂巢协作（关键）

1. 每个子代理任务必须声明：
- `scope`
- `non_goals`
- `allowed_files`
- `tests`
- `rollback`
- `commit_type`
2. 未声明上述字段，任务不得执行。
3. 子代理结果合并前必须通过策略检查（见 `engineering-policy.yaml`）。

## 9. 输出质量门槛

交付型输出默认包含：
1. 需求理解
2. 方案
3. 实现（优先 diff）
4. 使用示例
5. 测试
6. 自检清单

## 10. 自检清单（提交前）

1. 是否有重复代码
2. 是否有未处理异常
3. 是否有魔法数字
4. 是否可测试
5. 是否存在安全风险
6. 是否可读、可维护
