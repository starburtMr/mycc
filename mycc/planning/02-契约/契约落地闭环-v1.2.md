# 契约落地闭环（v1.2）

> 目的：把“契约先行”从文档约定变成可执行工程流程。

## 1. 目录与产物

1. `packages/contracts`
- 存放 zod schema、错误码枚举、公共类型。

2. `packages/client`
- 从 contracts 生成或封装 typed client（REST + SSE 参数与响应类型）。

3. `apps/backend`
- 只引用 contracts，不自行定义重复 DTO。

4. `apps/frontend`
- 只通过 client 访问接口，不手写裸请求类型。

## 2. 执行流程

1. 先改 contracts（schema + error code）。
2. 生成 OpenAPI 与 typed client。
3. 前后端分别实现。
4. 以契约测试校验字段一致性。

## 3. 强制校验

1. CI 必须检查：
- contracts 变更后是否更新 client。
- OpenAPI 是否与 schema 同步。

2. 禁止项：
- 后端新增未定义在 contracts 的响应字段。
- 前端新增未定义在 contracts 的请求字段。

## 4. 回滚策略

1. 契约变更使用版本号管理（v1.1 -> v1.2）。
2. 破坏性变更必须保留过渡期兼容层。
