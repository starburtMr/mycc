# 真源与重建规则（v1）

> 目标：消除“tasks/ 与 Event Store 双真源冲突”，统一恢复与对账口径。

## 1. 真源定义（拍板）

1. 状态真源：`SQLite.task_events`（事件账本）。
2. 文件真源：`tasks/<task_id>/runs/<run_id>/`（产物与 manifest）。
3. 快照层：`task_snapshot`（缓存，可丢，可重建）。

结论：
1. 任务状态以事件折叠结果为准。
2. 文件目录不直接承载状态机真相，只承载产物真相。

## 2. 重建规则

1. `GET /v1/tasks/:taskId`：
- 优先读 `task_snapshot`。
- snapshot 缺失/损坏 -> 从 `task_events` fold 重建。

2. `GET /v1/tasks/:taskId/events?after=<seq>`：
- 仅从 `task_events` 回放。
- 不从文件系统反推事件。

3. 产物校验：
- 以 `manifest.json` + 文件 hash 对账。
- manifest 中必须带 `event_seq` 与 `run_id`。

## 3. manifest 规范（最小字段）

```json
{
  "schema_version": 1,
  "task_id": "tsk_20260305_x1",
  "run_id": "run_20260305_01",
  "generated_at": "2026-03-05T10:00:00Z",
  "files": [
    {
      "path": "plan.md",
      "type": "markdown",
      "hash": "sha256:...",
      "size": 1024,
      "event_seq": 119,
      "created_at": "2026-03-05T10:00:10Z"
    }
  ]
}
```

## 4. 一致性检查

1. 事件一致性：`seq` 严格递增，无缺口（或缺口可回放补齐）。
2. 产物一致性：manifest 文件存在、hash 校验通过。
3. 快照一致性：snapshot 与 events fold 结果一致（抽样校验）。

## 5. 故障处理

1. snapshot 不一致：直接丢弃 snapshot，重新 fold。
2. manifest 缺失：标记 run 为 `warning`，触发修复任务。
3. 文件 hash 不一致：标记产物失效，不影响事件真源。

## 6. 验收口径

1. 任意时刻删除 snapshot 后，系统可从 events 自动恢复任务状态。
2. 任意 run 可通过 manifest + event_seq 回溯到来源事件。
3. UI 状态、事件回放、导出包三者对账一致。
