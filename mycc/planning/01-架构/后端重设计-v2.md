# MyCC 后端重设计规划（v2）

> 依据：对 `Aster110/mycc/.claude/skills/mycc/scripts` 的代码级复盘。
> 原则：保留可运行能力，去除路径耦合，升级为可恢复编排后端。

## 1. 复盘结论（从源码得到）

## 1.1 可借鉴能力

1. 启动自检流程完整：端口清理、CLI 检查、环境加载。
2. 通道抽象清晰：`ChannelManager` 支持多通道广播。
3. 隧道保活成熟：`TunnelManager` 有心跳、重试上限、防重入。
4. SDK 适配器实践有效：session 池、生命周期回调、多轮对话处理。
5. API 具备实用基础：`/pair`、`/chat`、`/events`、`/history` 等。

## 1.2 不适合直接复用的点

1. 入口文件过重（`index.ts` 职责过多）。
2. 路径耦合 `.claude/skills/...`，不适配 Claude+Codex 共治系统。
3. 关键状态以内存为主，重启恢复能力弱。
4. 缺少 run 维度的任务模型，不利于审计与回放。
5. 配对与鉴权对远程场景仍偏轻。

## 2. 设计目标（与现有系统对齐）

1. 对齐硬约束：EvoMap 前置、tasks 真源、route_reason 必记。
2. 支持双引擎编排：Claude/Codex 可统一路由。
3. 支持可恢复执行：任务、事件、运行实例可回放可追溯。
4. 保持单体可运行：先不拆微服务。

## 3. v2 逻辑架构

```text
API 层（Fastify）
  -> Orchestrator（PLAN/EXECUTE/EVAL/REFLECT）
    -> Adapter Router（claude/codex）
    -> Skill Router（skills-core registry）
    -> EvoMap Precheck Gate（只读）
    -> Memory Service（文件 + SQLite）
    -> Event Store（先落库后推流）
    -> Connectivity（local/lan/tunnel）
```

## 4. 模块拆分

## 4.1 API Gateway

职责：
1. 提供统一 REST + SSE。
2. 执行参数校验、鉴权、错误码标准化。
3. 仅负责协议，不承载业务决策。

## 4.2 Orchestrator

职责：
1. 管理任务状态机与阶段推进。
2. 维护 run 生命周期（start/cancel/retry/resume）。
3. 记录 `route_reason` 与验证结果。

## 4.3 Adapter Layer

职责：
1. 统一 Claude/Codex 输出事件协议。
2. 屏蔽 SDK 细节，暴露统一 stream 接口。
3. 标准化错误语义（retryable / fatal）。

## 4.4 Event Store

职责：
1. 事件先落库再推送 SSE。
2. 支持 `Last-Event-ID` / `after` 回放。
3. 维护 `seq` 单调递增与幂等。

## 4.5 Memory Service

职责：
1. 写入 working/fact/decision 三层记忆。
2. 保留 evidence_ref 追踪链。
3. 与文件系统保持索引一致性（path/hash）。

## 4.6 Connectivity

职责：
1. 默认 local。
2. 按需启用 lan/tunnel。
3. tunnel 只作为可插拔能力，不作为基础依赖。

## 5. 数据模型（v2 最小集合）

1. `tasks`：任务定义。
2. `task_runs`：每次执行实例。
3. `task_events`：事件账本（含 `seq`）。
4. `artifacts`：产物索引（path/hash/type/run_id）。
5. `pair_sessions`：配对挑战与时效。
6. `rate_limits`：配对与接口限流状态。

## 6. 接口策略（v2）

保留并升级：
1. `POST /v1/chat/send`
2. `POST /v1/chat/pair`
3. `POST /v1/tasks`
4. `POST /v1/tasks/:taskId/cancel`
5. `POST /v1/tasks/:taskId/retry`
6. `POST /v1/tasks/:taskId/resume`
7. `GET /v1/tasks/:taskId/events?after=`

## 7. 安全策略（MVP）

1. pair_code + challenge + TTL（60-120s）。
2. 失败锁定与限流审计。
3. SSE 连接鉴权与连接释放。
4. local/lan/tunnel 三模式权限分层。

## 8. 迁移策略（从现有系统演进）

1. 保留现有文档、skills-core、memory 资产不动。
2. 新后端落在 `mycc/backend` 独立目录。
3. 先接入契约与事件，再接引擎与 tunnel。
4. 完成后再替换旧入口，不做大爆炸式切换。

## 9. 分阶段实施建议

1. Stage A：API + 事件账本 + run 模型。
2. Stage B：双 adapter + route_reason 落库。
3. Stage C：EvoMap 前置闸门 + skill router 对接。
4. Stage D：配对强化 + tunnel 模块接入。

## 10. 验收标准（规划级）

1. 单任务可完整回放（创建到完成）。
2. 断线后可按 cursor 续流。
3. 任一执行都能追溯 route_reason。
4. 任一执行都能证明先 EvoMap 后执行。
