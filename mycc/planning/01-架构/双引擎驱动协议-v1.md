# 双引擎驱动协议（v1）

> 目标：明确后端如何稳定驱动 `Claude` 与 `Codex`，避免“只写了 adapter 名称，没有执行协议”。

## 1. 角色分工

1. Orchestrator：阶段推进与路由决策，不直接调用 SDK 细节。
2. Adapter Router：根据 `stage + route_policy + route_reason` 选择适配器。
3. `claude_adapter`：规划、解释、反思优先。
4. `codex_adapter`：执行、改代码、命令操作优先。

## 2. 统一输入协议（AdapterInput）

```json
{
  "task_id": "tsk_20260305_x1",
  "run_id": "run_20260305_01",
  "stage": "EXECUTE",
  "goal": "修复 SSE 回放重复问题",
  "constraints": ["先 EvoMap", "最小改动"],
  "context_refs": ["memory://fact/12", "task://last_failed_run"],
  "skills_allowlist": ["evomap-search", "code-review"],
  "route_reason": "EXECUTE 阶段需要本地代码执行能力"
}
```

## 3. 统一输出协议（AdapterEvent）

1. `delta`：文本增量
2. `tool_call`：工具调用意图
3. `tool_result`：工具结果摘要/引用
4. `final`：阶段输出总结
5. `error`：标准化错误（`retryable` 必填）

映射要求：
1. 所有事件必须映射到 `task_events`。
2. 事件写库后再推 SSE。
3. SDK 原始错误必须落成 `error` 事件，不允许透传堆栈到前端。

## 4. 路由策略

默认路由：
1. PLAN -> claude
2. EXECUTE -> codex
3. EVAL -> verifier（必要时 claude）
4. REFLECT -> claude

覆盖规则：
1. 明确要求“仅 Codex/仅 Claude”可覆盖默认路由。
2. 工具可用性不足时可切换备用适配器。
3. 覆盖必须写 `model_routed.route_reason`。

## 5. 生命周期与控制

1. `run()`：启动阶段执行流。
2. `cancel(run_id)`：中断适配器执行。
3. `timeout`：超时后上报 `error(code=TIMEOUT,retryable=true)`。
4. `retry`：新建 run，不复用旧 run。
5. `resume`：从最近 `run_checkpointed` 恢复。

## 6. 失败与回退

1. 适配器不可用：记录 `warning`，若允许则自动降级到另一个适配器。
2. 双适配器都不可用：任务置为 `failed`，写入可诊断信息。
3. 任何回退行为必须落 `model_routed` 与 `task_status`。

## 7. 验收标准

1. 同一任务可清晰追溯“为何选 Claude/Codex”。
2. 阶段失败时能区分：模型问题、工具问题、网络问题。
3. cancel/retry/resume 全链路可用且有事件证据。
4. 前端仅消费摘要事件也能完整理解任务进展。
