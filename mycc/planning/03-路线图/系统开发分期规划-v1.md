# MyCC 前后端系统开发分期规划（v1）

> 目的：把已确认共识固化为可执行路线图，作为后续前后端并行开发的统一基线。

## 0. 系统定义（本规划适用范围）

MyCC 要构建的是一个“协同智能体操作系统”：
1. 以聊天入口驱动任务执行，不割裂对话与工程执行。
2. 以后端编排中枢承接 Claude/Codex 协同与模型路由。
3. 以事件账本保障可回放、可审计、可恢复。
4. 以文件 + SQLite 记忆体系保障跨会话长期可用。
5. 以 skills-core 治理体系保障技能可控接入与可追踪执行。

明确不在本规划范围内：
1. 重型微服务化改造。
2. 默认公网暴露与默认 tunnel 常开。
3. 一次性覆盖所有终端形态的原生客户端。

## 1. 已确认的硬约束（不可漂移）

1. 架构目标：Claude + Codex 协同，不做单模型系统。
2. 记忆体系：全局采用 `文件 + SQLite` 双层记忆。
3. 任务真源：`tasks/` 是任务状态唯一真源。
4. 联网策略：执行前必须先做 EvoMap 经验检索；EvoMap 仅只读，不注册节点、不发 heartbeat、不接任务。
5. Skill 治理：统一标准目录 + registry 管理，走 `quarantine -> draft -> verified` 生命周期。
6. 连接策略：默认 `local`，仅在需要远程访问时启用 `tunnel`。
7. 文档与沟通：中文优先，关键决策必须落盘，所有路由必须记录 `route_reason`。

## 3. 分期路线图

## Phase 0：规划冻结与契约先行（P0）

目标：冻结前后端共同遵守的技术基线与 API 契约，确保可并行开发。

交付物：
1. 技术栈锁定文档（Node/TS、Next.js+Tailwind、Fastify、SQLite）。
2. API 契约草案（含聊天入口、任务入口、SSE 事件规范）。
3. 错误码与响应格式规范（统一 `error.code`、`request_id`、`details`）。
4. 事件模型规范（状态类/内容类/审计类）。

验收口径：
1. 前后端对字段命名、可选项、状态枚举完全一致。
2. 可以自动生成共享类型（避免文字契约漂移）。

## Phase 1：后端 MVP 骨架（P0）

目标：搭建可运行后端主干，先跑通请求、编排、事件流、落库。

交付物：
1. `backend` 服务可启动并提供健康检查。
2. 基础接口：
- `GET /v1/system/status`
- `POST /v1/tasks`
- `GET /v1/tasks/:taskId`
- `GET /v1/tasks/:taskId/events`
- `POST /v1/chat/send`
3. Orchestrator 四阶段：`PLAN -> EXECUTE -> EVAL -> REFLECT`。
4. Adapter 骨架：`claude_adapter`、`codex_adapter`（先空实现可替换）。
5. SQLite 建表与迁移入口（包含 `tasks/task_runs/task_events`）。
6. 入口拆分：启动流程、API 层、编排层、连接层分离，不再单文件承载。

验收口径：
1. 创建任务后可看到阶段事件流。
2. 事件可落库，任务状态可回查。
3. 不依赖 `.claude/` 路径即可运行。

## Phase 2：SSE 稳定性与可恢复能力（P0）

目标：解决流式场景下断线、丢事件、重复事件问题。

交付物：
1. 支持 `Last-Event-ID` 或 `after` 光标回放。
2. 事件序列号机制（`seq`）与客户端幂等去重策略。
3. 心跳与连接释放机制（断开自动清理）。
4. `cancel/retry/resume` 任务控制端点。
5. 事件写入顺序：先落库再推流。

验收口径：
1. 人工断开 SSE 后可从断点恢复。
2. 同一任务多次运行可追溯（run 维度清晰）。

## Phase 3：前端 MVP（P0）

目标：完成单入口交互（聊天 + 任务）并接通后端。

交付物：
1. `frontend` 项目初始化（Next.js + Tailwind）。
2. 页面骨架：
- 聊天入口页（默认入口）
- 任务详情页（阶段与进度）
- 系统状态页
- Debug 抽屉（route_reason、tool 调用、verification、memory_write）
3. SSE 订阅与断线重连。
4. Mock/真实接口可切换。

验收口径：
1. 前端可从聊天入口触发任务并实时显示事件。
2. 前端刷新后可回放任务状态与事件。

## Phase 4：系统规则落地（P1）

目标：把你已定义的系统原则变成后端硬闸门。

交付物：
1. EvoMap 前置检索闸门（失败阻断执行阶段）。
2. 路由记录闸门（每次模型选择都写 `route_reason`）。
3. Skill Router 与 registry 对接（只允许已注册可用技能执行）。
4. Memory 写入策略（working/fact/decision 分层，含 `namespace` 与 `evidence_ref`）。

验收口径：
1. 任一任务都能追溯“先检索再执行”的证据。
2. 任一任务都能追溯模型路由理由。

## Phase 5：联机与协作增强（P1）

目标：在稳定本地闭环后，补齐远程访问与知识沉淀。

交付物：
1. `local/lan/tunnel` 三模式切换策略与安全约束。
2. 移动端配对强化（短码时效、限流、审计）。
3. Obsidian 沉淀流程联动（会话总结自动产出到专属目录）。
4. 回归与演练脚本（备份恢复、断线恢复、任务重试）。
5. tunnel 作为可插拔模块，默认关闭，按需启用。

验收口径：
1. 本地与远程访问可切换且可控回滚。
2. 关键知识沉淀可自动化执行并可追踪。

## 4. 并行开发原则（前后端协作）

1. 先契约后实现：接口与事件 schema 未冻结前，不进入大规模编码。
2. 共享类型优先：禁止前后端手写重复类型定义。
3. 每日只做一次契约对齐，避免频繁返工。
4. 前端先跑 mock，再切真实接口；后端先保证稳定事件流再扩功能。

## 5. 明确不做（当前阶段）

1. 不上重型微服务拆分。
2. 不在 P0 追求全平台原生客户端。
3. 不默认启用 tunnel。
4. 不引入与当前目标无关的新依赖链。
