# 前端交互与信息架构（v1）

> 来源：`mycc/frontend/demo.html` 交互验证稿。  
> 目标：把可用 Demo 转成工程实现标准，避免后续页面重做。  
> 视觉与组件基线见：`05-前端体验/Agent-IDE设计系统-v1.md`。

## 1. 核心设计原则

1. 单入口：用户只从“聊天输入框”开始，不要求先选复杂模式。
2. 可追溯：聊天、任务、审计事件必须能相互跳转与对齐。
3. 可恢复：刷新或断线后，历史会话与任务状态可恢复。
4. 分层展示：主视图简洁，调试信息放右侧抽屉。
5. 功能边界：`Chat = 交互入口`，`Task = 执行对象`，`Run = 可恢复执行实例`，`Debug = 工程审计`。
6. 体验定位：`Agent IDE`，不是通用 dashboard，不做“日志淹没聊天”。

## 1.1 功能边界（强约束）

1. 聊天页只展示：
- 用户/助手对话
- Task 摘要卡（状态、阶段、进度、Open Task）
- 创建策略开关（Auto / Force Task / Chat Only）
- 默认行为：`Auto` 下优先纯聊天，仅在命中任务化条件时创建 task/run

2. 聊天页禁止展示：
- 工具调用明细（tool_called/tool_result）
- 路由候选明细（model_routed 候选表）
- 验证细节与记忆写入细节
- 全量事件流日志

3. 任务页必须承载：
- run 切换（retry/resume 后的多 run 视角）
- 阶段时间线、产物、验证、调试事件
- SSE 回放与连接状态

## 2. 页面信息架构

1. 左侧栏
- 页面导航：聊天工作台 / 任务详情 / Skills / 系统状态 / 记忆视图
- 聊天历史：会话列表（可搜索、切换、重命名、删除、新建、清空、重置）
- 最近任务：最近 run 状态快照

2. 中间主区
- 聊天工作台：消息流 + 输入区（发送 + 调度策略）；仅显示 Task 摘要卡
- 任务详情：任务状态卡、阶段卡、路由与事件游标卡、run 切换、动作区
- 系统状态：连接模式、预检状态、技能与真源状态
- 记忆视图：每条记忆卡片化展示

3. 右侧抽屉
- 审计事件卡：`model_routed`、`verification`、`memory_write`

## 3. 聊天历史能力（MVP 必须）

1. 会话列表本地持久化（MVP 用 localStorage，正式版接后端）。
2. 会话切换后，主区消息立即同步。
3. 支持搜索标题与消息内容。
4. 支持重命名、删除单会话、清空全部。
5. 空状态必须可读（无会话、无消息、无搜索结果）。

## 4. 卡片化规则（任务与记忆）

1. 任务详情页：
- 任务状态卡
- 路由决策卡（展示 `route_reason`）
- 事件游标卡（展示 `seq` 与回放点）
- 阶段卡（PLAN/EXECUTE/EVAL/REFLECT）

2. 记忆视图页：
- 每条记忆一张卡（`working_memory/fact_memory/decision_log`）
- 卡片至少显示：类型、核心值、`evidence_ref` 或过期时间

3. 卡片通用字段：
- `title`
- `summary/value`
- `meta`（时间、ref、状态）

## 5. 前端状态边界

1. 服务端状态（TanStack Query）：
- tasks、task snapshot、skills、system status

2. UI 状态（zustand）：
- 当前 activeSessionId
- 抽屉开关、筛选条件、输入草稿

3. SSE 事件处理：
- 进入 Query cache（或统一 event store）
- 按 `seq` 去重并更新快照

## 6. API 对应关系（前端视角）

1. 聊天发送：`POST /v1/chat/send`
2. 任务快照：`GET /v1/tasks/:taskId`
3. 任务流：`GET /v1/tasks/:taskId/events?after=<seq>`
4. 重试恢复：`POST /v1/tasks/:taskId/retry` / `resume`
5. 任务动作：`POST /v1/tasks/:taskId/cancel` / `export`

## 6.1 任务创建判定矩阵（聊天页）

1. `dispatch_mode=chat_only`
- 永不创建 task/run。
- 聊天仅返回助手文本结果。

2. `dispatch_mode=force_task`
- 必须创建 task/run。
- 聊天插入 Task 摘要卡，细节进入任务页。

3. `dispatch_mode=auto`（默认）
- 普通问答、解释、讨论：不创建 task/run。
- 命中以下条件之一才创建 task/run：
  - 明确执行意图（如“执行/落盘/产出文件/跑验证/修复并提交”）
  - 需要异步长任务（预计执行时间超过阈值）
  - 需要工具链与副作用操作（文件写入、脚本执行、网络调用）

## 7. 响应式与可用性

1. `<=980px` 时三栏降为单栏堆叠。
2. 抽屉移动到主区下方，避免遮挡输入。
3. 点击区域和字体尺寸保持移动可操作性。

## 8. 从 Demo 到正式实现

1. 先保留信息架构，不先改视觉皮肤。
2. 先把 localStorage 会话替换为后端会话接口。
3. 先接真实 SSE，再做细节动画与样式增强。
4. 视觉实现阶段必须遵循 `Agent-IDE设计系统-v1.md` 的 token 与组件语义。
