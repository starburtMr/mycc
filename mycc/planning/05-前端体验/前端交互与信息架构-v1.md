# 前端交互与信息架构（v1）

> 来源：`mycc/frontend/demo.html` 交互验证稿。  
> 目标：把可用 Demo 转成工程实现标准，避免后续页面重做。

## 1. 核心设计原则

1. 单入口：用户只从“聊天输入框”开始，不要求先选复杂模式。
2. 可追溯：聊天、任务、审计事件必须能相互跳转与对齐。
3. 可恢复：刷新或断线后，历史会话与任务状态可恢复。
4. 分层展示：主视图简洁，调试信息放右侧抽屉。

## 2. 页面信息架构

1. 左侧栏
- 页面导航：聊天工作台 / 任务详情 / 系统状态 / 记忆视图
- 聊天历史：会话列表（可搜索、切换、重命名、删除、新建、清空、重置）
- 最近任务：最近 run 状态快照

2. 中间主区
- 聊天工作台：消息流 + 统一输入区（发送、转任务）
- 任务详情：任务状态卡、阶段卡、路由与事件游标卡
- 系统状态：连接模式、预检状态、技能与真源状态
- 记忆视图：每条记忆卡片化展示

3. 右侧抽屉
- 审计事件卡：`model_routed`、`verification`、`memory_write`

## 3. 聊天历史能力（MVP 必须）

1. 会话列表本地持久化（MVP 用 localStorage，正式版接后端）。
2. 会话切换后，主区消息立即同步。
3. 支持搜索标题与消息内容。
4. 支持重命名、删除单会话、清空全部。
5. 空状态必须可读（无会话、无消息、无搜索结果）。

## 4. 卡片化规则（任务与记忆）

1. 任务详情页：
- 任务状态卡
- 路由决策卡（展示 `route_reason`）
- 事件游标卡（展示 `seq` 与回放点）
- 阶段卡（PLAN/EXECUTE/EVAL/REFLECT）

2. 记忆视图页：
- 每条记忆一张卡（`working_memory/fact_memory/decision_log`）
- 卡片至少显示：类型、核心值、`evidence_ref` 或过期时间

3. 卡片通用字段：
- `title`
- `summary/value`
- `meta`（时间、ref、状态）

## 5. 前端状态边界

1. 服务端状态（TanStack Query）：
- tasks、task snapshot、skills、system status

2. UI 状态（zustand）：
- 当前 activeSessionId
- 抽屉开关、筛选条件、输入草稿

3. SSE 事件处理：
- 进入 Query cache（或统一 event store）
- 按 `seq` 去重并更新快照

## 6. API 对应关系（前端视角）

1. 聊天发送：`POST /v1/chat/send`
2. 任务快照：`GET /v1/tasks/:taskId`
3. 任务流：`GET /v1/tasks/:taskId/events?after=<seq>`
4. 重试恢复：`POST /v1/tasks/:taskId/retry` / `resume`

## 7. 响应式与可用性

1. `<=980px` 时三栏降为单栏堆叠。
2. 抽屉移动到主区下方，避免遮挡输入。
3. 点击区域和字体尺寸保持移动可操作性。

## 8. 从 Demo 到正式实现

1. 先保留信息架构，不先改视觉皮肤。
2. 先把 localStorage 会话替换为后端会话接口。
3. 先接真实 SSE，再做细节动画与样式增强。
